# logs/ — 过程日志规范

## 为什么写日志

日志是写给"未来的自己"看的。假设读日志的人完全不知道之前发生了什么，也能通过日志还原整个过程。

## 文件命名

```
YYYYMMDD_HHMM_简短主题.md
```

示例：`20260602_1430_原始架构分析.md`

## 日志格式

```markdown
# [简短标题]

## 时间
YYYY-MM-DD HH:MM

## 做了什么
[具体操作]

## 为什么这样做
[理由和依据]

## 结果
[关键发现/输出]

## 下一步
[计划做什么]

## 备注
[发现的问题、潜在风险、需要确认的点]
```

## 规范要点

- **每步都写**：拆题、建模、编码、验证、每轮迭代，都留日志
- **结论自洽**：如果后来的日志推翻了前面的结论，在备注里说明
- **不说废话**：记录决策原因比记录代码细节更重要
- **嵌入图表**：关键结果截图或图表直接贴入，方便回顾
